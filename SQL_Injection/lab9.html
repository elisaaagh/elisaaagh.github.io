Lab 9 - Blind SQL injection with conditional responsess

Nivel

Practitioner

Tipo de SQLi

Inyección ciega booleana

Objetivo

Explotar una inyección SQL ciega para deducir la contraseña del usuario administrator.

Explicación técnica de la vulnerabilidad

La aplicación utiliza el valor de la cookie de rastreo (TrackingId) dentro de una consulta SQL sin la debida sanitización. Aunque los resultados de la consulta no se muestran en pantalla, la aplicación exhibe un comportamiento condicional: si la consulta inyectada evalúa como verdadera, el servidor incluye el mensaje "Welcome back" en la respuesta HTTP, si evalúa como falsa, el mensaje se omite. Esta diferencia booleana permite inferir datos de la base de datos haciendo preguntas de si o no.

Procedimiento paso a paso

  • Interceptar la petición que contiene la cookie TrackingId y enviarla a Repeater en Burp Suite.
  • Probar una condición verdadera y una falsa para confirmar la vulnerabilidad de SQL Injection observando el mensaje "Welcome back".
  • Verificar la existencia de la tabla users y del usuario administrator mediante consultas SQL inyectadas.
  • Determinar la longitud de la contraseña usando consultas con LENGTH(password).
  • Enviar la petición a Intruder para descubrir la contraseña carácter por carácter usando SUBSTRING().
  • Configurar los payloads con caracteres alfanuméricos y detectar la respuesta correcta mediante el texto "Welcome back".
  • Repetir el proceso para cada posición hasta obtener toda la contraseña.
  • Iniciar sesión en My account con el usuario administrator y la contraseña obtenida.

Explicación de payload

TrackingId=TuCookie' AND (SELECT SUBSTRING(password,1,1) FROM users WHERE username='administrator')='a.

  • La comilla simple (') cierra la cadena original del TrackingId.
  • El operador AND permite agregar una condición lógica a la consulta SQL.
  • La subconsulta SELECT SUBSTRING(password,1,1) extrae el primer carácter de la contraseña del usuario administrator.
  • La comparación ='a' verifica si ese carácter coincide con el valor probado.
  • Si la condición es verdadera, la aplicación responde normalmente y aparece el mensaje "Welcome back", confirmando el carácter correcto.

Mitigación recomendada

La solución definitiva es implementar Consultas Parametrizadas (Prepared Statements) en el código del servidor. Esto garantiza que la entrada proporcionada por el usuario (en este caso, la cookie TrackingId) se trate estrictamente como un valor de tipo cadena (string) y nunca como código SQL ejecutable, previniendo así cualquier alteración de la lógica de la base de datos.
